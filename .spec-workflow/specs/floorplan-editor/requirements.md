# Requirements Document — 間取りエディタ

## Introduction

間取りエディタは、ユーザーが自宅の間取りをアプリ上に再現し、掃除対象となるエリア（部屋）と家具・家電を配置するための機能。ここで作成した間取りがヒートマップ表示・掃除記録の土台となるため、本アプリの起点となる中核機能である。

操作は「部屋を追加 → 部屋内に家具・家電を配置」の2段階構造とする。住所・位置情報は一切扱わない。

## Alignment with Product Vision

- **視覚的直感性**（Product Principle 4）: テキスト入力ではなく、矩形のドラッグ・グリッド配置という視覚操作で間取りを構築する
- **シンプルファースト**（Product Principle 1）: 部屋の形状を矩形に限定し、グリッドスナップで整列を自動化することで操作を最小化する
- **プライバシー優先**（Product Principle 2）: 間取りは抽象的な矩形配置のみで、住所・実寸・GPSを保存しない
- **間取り設定完了率 > 70%**（Success Metrics）: オンボーディングで挫折しない軽量な編集体験を目指す

## Requirements

### Requirement 1: 部屋の追加・配置

**User Story:** ユーザーとして、間取り図に部屋を追加して配置したい。掃除対象のエリアを画面上で把握できるようにするため。

#### Acceptance Criteria

1. WHEN ユーザーが「部屋を追加」を実行する THEN システムは グリッドにスナップした矩形の部屋をキャンバスに配置する SHALL
2. WHEN ユーザーが部屋をドラッグする THEN システムは 最も近いグリッドセルに部屋をスナップして移動する SHALL
3. WHEN ユーザーが部屋の端をドラッグする THEN システムは グリッド単位で部屋の幅・高さをリサイズする SHALL
4. IF 配置・リサイズ後の部屋が他の部屋と重なる THEN システムは 重なりを視覚的に警告する（配置は許可する）SHALL
5. WHEN ユーザーが部屋に種別（キッチン・バスルーム等）を設定する THEN システムは 種別に応じたプリセットパーツを seed として部屋に紐付ける SHALL
6. WHEN ユーザーが部屋を削除する THEN システムは 確認のうえ、その部屋に紐付く家具・パーツ・掃除記録もまとめて削除する SHALL

### Requirement 2: 家具・家電の配置（部屋内）

**User Story:** ユーザーとして、部屋の中に家具や家電を配置したい。家具単位でも掃除状態を管理できるようにするため。

#### Acceptance Criteria

1. WHEN ユーザーが部屋を選択して「家具を追加」を実行する THEN システムは プリセット家具一覧を提示する SHALL
2. WHEN ユーザーがプリセットにない家具を追加したい THEN システムは 自由名称で家具を作成できる SHALL
3. WHEN ユーザーが家具をドラッグする THEN システムは グリッドにスナップして所属する部屋の内部に家具を配置する SHALL
4. IF 家具が所属部屋の矩形外に出る THEN システムは 部屋の境界内に収まるよう配置を制約する SHALL
5. WHEN ユーザーが家具を削除する THEN システムは その家具に紐付くパーツ・掃除記録もまとめて削除する SHALL

### Requirement 3: 間取りの永続化

**User Story:** ユーザーとして、作成した間取りを保存して次回起動時にも使いたい。毎回作り直さずに済むようにするため。

#### Acceptance Criteria

1. WHEN ユーザーが間取りを編集する THEN システムは 変更をサーバー（PostgreSQL）に保存する SHALL
2. WHEN アプリを再起動する THEN システムは サーバーから間取りを取得して復元する SHALL
3. IF サーバーへの保存が失敗する THEN システムは ユーザーに失敗を通知し、再試行手段を提供する SHALL
4. WHEN 間取りを取得する THEN システムは 初回起動時に発行したUUIDに紐付く間取りのみを返す SHALL

### Requirement 4: 初回オンボーディング

**User Story:** 新規ユーザーとして、最初に間取りをスムーズに作成したい。アプリの価値を体験する前に挫折しないため。

#### Acceptance Criteria

1. WHEN ユーザーが初めてアプリを起動する THEN システムは 間取りエディタへ誘導する SHALL
2. WHEN 間取りが1つも存在しない THEN システムは 部屋追加を促す空状態（empty state）を表示する SHALL
3. WHEN ユーザーが最初の部屋を配置する THEN システムは 種別選択を促し、プリセットパーツの恩恵を即座に得られるようにする SHALL

## Non-Functional Requirements

### Code Architecture and Modularity

- structure.md の Package by Feature 構成に従い、`features/floorplan/` 配下に components/hooks/usecases/repositories を分離する
- 描画（components）にビジネスロジックを持たせず、グリッドスナップ計算等は usecases または shared/utils の純粋関数に置く
- 他featureがこの間取り情報を参照する場合は `FloorplanCapability` 経由とし、直接importを禁止する

### Performance

- ヒートマップ仕様と整合し、100エリアを含む間取りでも 60fps の描画・ドラッグ操作を維持する
- グリッドスナップ計算はドラッグ中に毎フレーム呼ばれるため、純粋関数として軽量に保つ

### Security

- 住所・実寸・GPS座標は保存しない。間取りはグリッド座標（相対値）のみで表現する
- 間取りデータは初回発行UUIDに紐付け、他ユーザーのデータにアクセスできない

### Reliability

- サーバー保存失敗時もユーザーの編集内容を即座に失わせない（再試行できる）

### Usability

- 部屋・家具の配置は3タップ以内を目指し、グリッドスナップで整列を自動化する
- ダークモード（システム設定追従）に対応する
